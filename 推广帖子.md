# 学了一圈 Agent，我好像才反应过来手写最值

说真的，一开始我就是照着教程调调 API，十几分钟搭出来一个 Agent，当时心里还嘀咕：哎，这玩意儿好像也不难嘛。

但等我真想让它有点多轮记忆、能自己在几个工具里挑着用、甚至好几个 Agent 凑一块儿干活的时候——全乱套了。就是那种你说不清哪儿不对，但它就是不对。我想了半天，感觉最根本的问题可能就一个：我压根不知道这 Agent 脑子里在想啥。

后来我就反过来，干脆把 ReAct 自己手撸一遍。

## 不手写一遍，你真不知道框架替你藏了多少东西

ReAct 这东西，大白话讲就是个死循环：想一下 → 动一下 → 看结果 → 接着想。但你用框架的时候，这一整套是黑盒，它为啥偏偏选了这个工具、参数为啥这么传，你一无所知。

我就拿最原始的 prompt 加原生 API，硬把这个循环给转起来：

```python
while not done:
    thought = llm("分析情况，决定下一步")  # Thought
    action = parse(thought)                 # Action
    obs = run_tool(action)                  # Observation
    # 带着 obs 进下一轮，直到给最终答案
```

写完那一下我有点开窍（可能也没那么窍吧，但至少不那么懵了）——框架那个 `create_agent`，扒了皮看，不就是上面这破循环嘛。无非是帮你把解析、调度、拼 prompt 这些脏活累活都包了。

**自己手写过一遍，再扭头看框架，怎么说呢，到处都透着光。**

## RAG 其实是同一回事

我以前一直觉得 RAG 不就是「文档切吧切吧 → 向量找一找 → 塞 prompt 里」嘛，能有多难。等自己上手写才知道坑有多深：chunk_size 你随便那么一刀切下去，召回效果能天差地别；混合检索为啥就比纯向量香；还有那个 HyDE 改写，它到底改了个啥东西……

这些东西吧，我觉得光看文档真没用。你非得自己动手，改个参数，跑一遍，死盯着输出看变化，那点"直觉"才能慢慢长出来。就好比学做菜，光看菜谱你永远不知道火候是啥手感。

## 多智能体，我翻车翻得最惨的地方

单个 Agent 玩溜之后，我就琢磨着搞几个 Agent 一起协作——结果直接卡死。它们仨状态咋传？谁指挥谁？排队还是一块儿上？我想了好几天也没太想明白。

这块我耗的时间最长，最后勉强捋出来几种套路（也不一定全对哈，但至少能跑通）：

- **supervisor**：中间坐个"领导"，看情况往谁那儿派活（有点中心化那个意思）
- **swarm + handoff**：Agent 之间互相甩锅……不是，是互相交接（去中心化）
- **并行 map-reduce**：把一个事儿拆成好几摊，一起跑完再收拢回来

最搞人的是，同一件事，你拿 LangGraph、CrewAI、AutoGen 写出来完全是三个画风。我之前一直陷在 LangGraph 里出不来，后来把三个放一块儿对比着看，才慢慢有点感觉。光盯一个框架死磕，真的容易钻牛角尖。

## 我把这些折腾，整理了一下

整个过程我慢慢攒成了四门课，每一节基本就是「手写原理 + 能跑的代码 + 练习」这么个路子：

- RAG 手写（9 节）：从 embedding 一路干到工程化
- Agent 手写（9 节）：从 Function Calling 玩到多智能体
- 框架进阶（9 节）：LangChain / LangGraph，每课都是手写版对打框架版
- 多智能体编排（9 节）：六种拓扑结构 + 三个框架横向对比

学完有个毕业设计——一个多智能体的并行研究系统，能联网搜东西、并行做检索、还能审稿迭代，最后 Docker 一打包就能跑。算是把我这一路学的东西东拼西凑缝起来的一个东西吧，有点缝合物那意思。目前这个项目单独搁在 `feat/portfolio-research-assistant` 分支上，**还在改**，还没并回主干。
