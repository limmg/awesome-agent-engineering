# 复现笔记：多 Agent 记忆共享

> Frontier L12 最小 idea 复现。诚实记录 idea / 实现取舍 / 结果 / 与论文差异。

## Idea

**假设**：多个 Agent 协作时，共享一个记忆库（Agent A 的发现 Agent B 能 recall），比各自独立记忆，信息复用率更高。

**来源**：这是 Agent 多智能体协作 + 记忆系统交叉方向的常见假设。没有单一论文对应，而是 Generative Agents（Park et al. 2023）的记忆共享思想 + 多 Agent 协作框架（AutoGen/CrewAI）的实践方向综合。

## 实现取舍

| 取舍 | 选择 | 理由 |
|------|------|------|
| 记忆检索 | 关键词匹配（非 embedding） | 最小复现原则——去掉非必要工程 |
| Agent 数量 | 2 个（A/B） | 最小可验证 |
| Agent 能力 | 预设发现（非真实 LLM） | 不依赖 API，验证机制而非模型能力 |
| 记忆库 | dict（非 Chroma） | 最小复现，不用外部依赖 |
| 指标 | 信息复用率（B 是否引用 A） | 最直接的"共享有没有用"信号 |

**核心简化**：用预设发现代替真实 LLM 研究。这意味着**验证的是"共享机制本身能否传递信息"，不是"真实多 Agent 协作的质量"**。

## 结果

| 模式 | B 引用 A 的发现？ | 信息复用率 |
|------|------------------|-----------|
| 独立记忆 | ❌ 否 | 0% |
| 共享记忆 | ✅ 是 | 100% |

**复现成功**：共享记忆模式下，Agent B 确实 recall 到了 A 的发现并引用。独立模式下 B 完全看不到 A 的发现。

## 与"论文"（假设）的差异

1. **真实性差距**：预设发现 ≠ 真实研究。真实场景下 Agent B 是否**愿意**引用 A 的发现，取决于任务相关性——如果 B 的主题和 A 完全无关，共享记忆也没用（recall 不到相关的）。
2. **噪音问题**：本实验只有 2 个 Agent，记忆库里就 1 条。真实场景 N 个 Agent × M 次研究 = 记忆库很大，recall 会被噪音淹没（需要 L01 的遗忘策略 + L02 的反思式提炼配合）。
3. **冲突问题**：如果 A 和 B 的发现矛盾呢？共享记忆会把矛盾传给 B——这时需要 L05 的冲突检测机制。本实验没测这个。

## 结论

**核心 idea 验证成立**：共享记忆确实让信息跨 Agent 流动（复用率 0→100%）。

**但真实价值需要更多验证**：
- 多 Agent（>2）时记忆库噪音问题
- 任务不相关时共享是否有用（可能反而干扰）
- 冲突发现时的处理

**下一步**：如果深入，会接入 L01 的 MemoryStore（Chroma + embedding）替换 MiniMemory，用真实 LLM 替换预设发现，测 3+ Agent 的协作。

## 复现成本

- 代码：~180 行（含注释和实验框架）
- 时间：约 1 小时（含设计对照实验）
- 依赖：零（纯 Python 标准库）

## 一句话总结

> 多 Agent 记忆共享的**机制**成立（信息能跨 Agent 流动），但**真实价值**取决于任务相关性 + 记忆库质量管理——不是"共享了就好"，还要解决噪音和冲突。
