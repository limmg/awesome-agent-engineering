## 预期模式（LLM 服务的典型压测曲线）

基于对 kb-qa 架构的分析（检索 ~300-500ms + 生成 ~1.5-3s，瓶颈在智谱 API）：

```
QPS
 │        ╭──── 天花板（上游限流，QPS 不再涨）
 │      ╱
 │    ╱
 │  ╱     ← 容量内线性增长
 │╱
 └──────────────── 并发数
                  ↑
              拐点（错误率开始上升处）
```

- **并发 1~N**：QPS 随并发近似线性增长，P95 稳定（~2-3s）→ 容量内
- **并发 >N**：QPS 停滞，P95 飙升（排队）→ 容量饱和
- **并发 >>N**：错误率上升（智谱 429）→ **这是天花板**

`N`（拐点）取决于智谱 API 的并发限额，通常个位数到十几。

## 解读

- **QPS 随并发线性增长到某点后停滞** → 那点就是吞吐天花板
- **P95 在某并发档突然飙升** → 容量饱和，排队严重
- **错误率上升（通常 429）** → 上游 API 限流，是 LLM 服务的主要瓶颈
- **应对**：用 `asyncio.Semaphore` 把生产并发卡在拐点以内，超过的快速拒绝，
  保护上游不被全面封禁（research-assistant 的 `_search_semaphore` 同思路）

## 关键认知

**LLM 服务的瓶颈通常在智谱 API 限流而非本地 CPU**。很多团队压 LLM 服务发现 QPS 上不去
就去优化本地代码（多线程/缓存查询），但瓶颈在上游，本地优化无效。
正确做法：本地信号量限流 + 语义缓存（L10 减少命中重复调用）+ 模型选型（L12 用 flash 替代）。
